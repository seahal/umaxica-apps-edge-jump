# ADR 0001: Do Not Serve Static Assets From Jump

## Status

Accepted

## Context

Jump is an edge HTTP service that runs on Fastly Compute and Cloudflare Workers.
It brokers redirect trust across FQDN boundaries and intentionally keeps the
runtime path small, stateless, and framework-independent.

The service currently returns a few stable responses, such as `/robots.txt`,
`/about`, `/health`, and `/.well-known/jwks.json`, from application routes. These
responses are generated by the Worker/Compute application code. They are not
served from a `public/` directory, Workers Assets, Cloudflare Pages, R2, or a
Fastly static file asset pipeline.

This distinction matters because Vercel-style `public/` semantics are useful for
frontend applications, but Jump is not a frontend app and does not need favicon,
manifest, Open Graph image, or SPA fallback behavior.

## Decision

Jump will not provide static asset serving in this area.

Specifically:

- Do not add a Vercel-style `public/` asset contract for Jump.
- Do not configure Cloudflare Workers Assets for Jump unless a future ADR
  changes this decision.
- Do not configure Cloudflare Pages output directories for Jump.
- Do not use R2 as an asset bucket for Jump-owned public files.
- Do not add Fastly package-managed static files for favicon, manifest, or OG
  images.
- Do not proxy static asset requests through the Worker/Compute application.
- Keep stable HTTP responses, such as `/robots.txt`, as explicit application
  routes when they are needed for the service contract.

## Consequences

The service avoids asset pipeline coupling across Fastly and Cloudflare. The
Worker/Compute application remains responsible for redirect handling, health
checks, robots policy, and well-known JSON responses only.

Requests for frontend-style assets such as `/favicon.ico`,
`/manifest.webmanifest`, `/sitemap.xml`, and Open Graph images are not part of
the Jump service contract. If noise from `/favicon.ico` becomes operationally
relevant, Jump may add a small explicit application route such as `204 No
Content`; that is still not a static asset pipeline.

`Cache-Control: no-store` remains appropriate for current Jump application
responses. Immutable CDN caching, content-type inference for static files,
SPA fallback rules, `_headers`, `_redirects`, and OpenNext/Next.js public asset
handling are outside the scope of this service unless Jump is later re-scoped as
a frontend application.

## Future Review

Revisit this decision only if Jump starts owning a browser-facing frontend or a
separate requirement appears for public binary assets. At that point, the new ADR
must choose one asset owner and deployment path instead of mixing Cloudflare
Workers Assets, Cloudflare Pages, Fastly package assets, R2, and framework
public directories.
